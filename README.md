# Masato - Pasada

> *"Know when to ride. Know when to leave."*

A demand-first public transport coordination platform for jeepney cooperatives.

**Hackathon:** SparkFest 2026 — GDG on Campus PUP  
**SDG:** 11 — Sustainable Cities & Communities  
**Deadline:** July 2, 2026  
**Repo:** https://github.com/sjmbaldesco/masatohackathon

---

## Problem

The Philippine jeepney system runs on a two-sided waiting problem. Passengers wait blindly at the curb with no ETA. Drivers idle at terminals until the jeep fills, blind to where passengers are. Cooperatives have no dispatch data. Both sides lose time. Low-income commuters absorb all the inefficiency.

**This is not a technology gap. It is an information gap.**

---

## Solution

Pasada digitizes the street-level demand that already exists. Passengers don't *book* a jeep — they *broadcast* that they're waiting. Drivers see live aggregated demand and get an AI confidence score on whether to depart now. Cooperatives see the full fleet picture and export analytics. No ride-hailing, no booking, no legal complications.

## What Is Built

### Passenger
- Select boarding stop on live Google Map (gray style, R01 route polyline)
- Tap **Signal** to broadcast waiting status — button toggles to rust "Waiting at {Stop}"
- Signal persists across page refresh (Firestore restore on mount)
- Live ETA countdown as the nearest jeep approaches
- Person SVG marker for own GPS position
- Recenter button (snaps map back to GPS location)

### Driver
- Live demand heatmap — two concentric circles per stop, color-coded by count:
  - Blue (≤2) → Teal (≤4) → Orange (≤7) → Red (>7)
- Animated jeep marker at 60fps (rAF ease-out interpolation between 500ms Firestore ticks)
- **Departure Confidence Score** — deterministic base score (fill ratio power curve + idle bonus) with Gemini ±10 adjustment, 30s cache
- Update Occupancy bottom-sheet (SVG ring, stepper)
- **Start Trip** button launches haversine simulation along ORS-generated polyline
- Recenter button (follows jeep position)

### Admin (Cooperative Dashboard)
- **Live Ops tab:** Full R01 route polyline, demand heatmap, per-unit status cards
- **Analytics tab:** Fleet KPIs (active drivers, total waiting, avg occupancy)
- **Export to Excel** — 4-sheet workbook: Summary, Drivers, Demand, AI Insights
- AI Insights sheet generated by Gemini via `/ai/analytics/insights` endpoint

### Core Infrastructure
- Firebase Auth: email/password (passenger, admin), synthetic email + PIN (driver)
- Firestore real-time via `onSnapshot` — all maps update live
- FastAPI backend with Firebase Admin token verification on all routes
- Gemini `gemini-1.5-flash` for Departure Score adjustment and Analytics Insights
- Route R01: Lumban → Sta. Cruz, Laguna — 10 stops, 223-point ORS-generated polyline
- ErrorBoundary catches all render-phase crashes without white-screening

## Firestore Schema

```
drivers/{uid}
  uid, driver_name, plate, route ("R01"), capacity (18)
  occupancy_count, occupancy_pct, speed_kmh
  lat, lng, current_stop, status ("idle"|"in_transit"), last_updated

passengers/{uid}
  route, status ("waiting"|"boarded"), lat, lng, stop

routes/{id}
  route_id ("R01"), name, polyline ([{lat, lng}] — 223 points)

users/{uid}
  uid, role ("passenger"|"driver"|"admin"), createdAt
```

## Running Locally

### Prerequisites
- Node.js 20+
- Python 3.12+
- Firebase project with Firestore + Auth enabled
- Google Maps Platform API key
- Gemini API key
- `backend/serviceAccountKey.json` (Firebase Admin SDK)

### Backend

```powershell
cd backend
.\venv\Scripts\Activate.ps1        # Windows — activate venv first
pip install -r requirements.txt
# Copy .env and fill in: GEMINI_API_KEY, FIREBASE_PROJECT_ID, ALLOWED_ORIGINS
uvicorn app.main:app --reload --port 8000
```

### Frontend

```powershell
cd frontend
npm install
# Copy .env and fill in: VITE_FIREBASE_* keys, VITE_MAPS_API_KEY
npm run dev                        # http://localhost:3000
```

> **Note (Windows):** The repo path contains a space and dash (`OneDrive - The Macabuhay`). Always quote paths in PowerShell. Activate the Python venv before running uvicorn.

## Demo Flow

1. **Admin** logs in → Live Ops shows R01 route and (initially empty) demand map
2. **Passenger** logs in → taps **Signal** at Lumban → demand circle appears on all views
3. **Driver** logs in → sees demand circle → Departure Score auto-fetches
4. Driver updates occupancy to 100% → Passenger sees "Full", Admin unit turns red
5. Driver taps **Start Trip** → jeep animates along route → Passenger ETA counts down
6. Admin → Analytics → **Export to Excel** → 4-sheet workbook with AI narrative
